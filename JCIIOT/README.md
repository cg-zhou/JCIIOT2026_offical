# Contestant Manual — Guide to Modifiable Files and Parameters

## Competition Introduction

This competition is based on a VLA (Vision-Language-Action) embodied intelligence platform built with **MuJoCo + robosuite + robomimic**. Contestants need to control a Tiago mobile manipulator robot in FactorySorting simulation scenes to complete transport tasks: grasping materials from a specified input station and moving them to a specified output station.

### Competition Tasks

The competition contains **five levels, L1 to L5, with increasing difficulty**. Each level corresponds to a different simulation scene and task description:

| Level | Scene           | Full Score |
| ---- | --------------- | ---------- |
| L1   | FactorySorting1 | 10         |
| L2   | FactorySorting3 | 15         |
| L3   | FactorySorting5 | 20         |
| L4   | FactorySorting7 | 25         |
| L5   | FactorySorting9 | 30         |

### Scoring Mechanism

Scores are automatically calculated by the built-in `_score_steps()` function in `app.py`. The core logic is as follows:

- **Leaving the source station** (successful grasp and the object moves more than 1 m away from the input station) — 50% of the total score
- **Reaching the target station** (successful grasp and the object is within 0.8 m of the output station table) — 50% of the total score
- **Collision penalty**: collision during navigation results in **-5 points**

> **Note**: Both scoring items require a successful grasp as a prerequisite. If the object is not successfully grasped, the score is directly set to 0.

After contestants click the "Execute" button in `app.py`, the system automatically runs the task, records the trajectory, and calculates the score.

### What Contestants Need to Do

1. **Understand the SOP**: Use a large model to automatically extract the Standard Operating Procedure from the Word documents in the `sop+prompt/` directory and generate knowledge-base `.md` files.
2. **Optimize skill code**: Modify the robot operation logic under `src/robot_agent/skills/` to improve the success rates of navigation, grasping, and placement.
3. **Tune execution parameters**: Adjust navigation speed, grasping steps, placement accuracy, and other parameters through `knowledge/robot_params.json`.
4. **Advanced: train your own BC model**: Collect demonstration trajectories, train a robomimic Behavior Cloning policy, and replace the default grasping model.

### Overall Workflow

```text
Execute prompt instruction -> LLM decomposes it into steps -> Agent executes each step -> Skill drives the simulation
```

## Competition Rules

### Permission to Modify Code and Configuration

- ✅ **Allowed to modify**: `src/robot_agent/skills/`, `src/robot_agent/workflows/`, `knowledge/robot_params.json`
- ❌ **Forbidden to modify**: `src/robot_agent/core/`, `src/robot_agent/environments/`, `app.py`, `knowledge/task_config.json`
- After modifying the code, run `app.py` and click "Execute" to view the score.

### SOP Knowledge-Base Rules (Important)

Files such as `sop1.md` to `sop5.md` and `sop_main.md` in the `knowledge/` directory are **reference versions** manually rewritten after reading the Word documents.

**Competition requirements:**

1. The original SOP documents are the five Word files in the `sop+prompt/` directory:

   | Word File                                  | Corresponding Level |
   | ------------------------------------------ | ------------------- |
   | `sop+prompt/JCIIOT 2026 case 1 SOP.docx`   | L1                  |
   | `sop+prompt/JCIIOT 2026 case 3 SOP.docx`   | L2                  |
   | `sop+prompt/JCIIOT 2026 case 5 SOP.docx`   | L3                  |
   | `sop+prompt/JCIIOT 2026 case 7 SOP.docx`   | L4                  |
   | `sop+prompt/JCIIOT 2026 case 9 SOP.docx`   | L5                  |

2. Contestants **must write their own code** (for example, using the `.docx` parsing and VLM visual analysis capabilities in `skills/read_document.py`) so that the large model can read the Word files above and automatically generate `.md` knowledge-base files that are **functionally equivalent** to the manually rewritten versions.
3. **Directly reusing the existing `sop*.md` files in the `knowledge/` directory will lead to point deductions**. Judges will compare the files generated by contestants with the manually rewritten versions to determine whether they are original AI-generated outputs.
4. Contestants may place the newly generated `.md` files in the `knowledge/` directory by overwriting old files or creating new ones. However, the code used for generation must be kept in `skills/` or `workflows/` for judges to review.

### Scene Map Resources

The project provides complete object position and size information for the five scenes. Contestants may use these data for map construction and navigation optimization.

**Original scene data** — `robosuite/robosuite/environments/factory_sorting/maps/`

| File                                                      | Corresponding Level |
| --------------------------------------------------------- | ------------------- |
| `factory_sorting_1_3fo3erfhisem_scene_regenerated.json`   | L1                  |
| `factory_sorting_3_3fo3errph7x9_scene_regenerated.json`   | L2                  |
| `factory_sorting_5_3fo3ertpxeut_scene_regenerated.json`   | L3                  |
| `factory_sorting_7_3fo3erfky9rn_scene_regenerated.json`   | L4                  |
| `factory_sorting_9_3fo3ert2c5fp_scene_regenerated.json`   | L5                  |

Each JSON file contains:

| Field        | Content                                                                                                          |
| ------------ | ---------------------------------------------------------------------------------------------------------------- |
| `bounds`     | Map boundaries (`x_min`, `x_max`, `y_min`, `y_max`)                                                              |
| `objects[]`  | Names, center coordinates, dimensions (length and width), orientation, type (station/obstacle), role (input/output), and production-line ownership of all objects |
| `robot`      | Initial robot pose                                                                                               |
| `resolution` | Grid resolution (meters per cell)                                                                                |

**Pre-generated maps** — `robosuite/robosuite/environments/factory_sorting/generated_maps/`

Each scene corresponds to three files, automatically generated by the `get_map.py` tool:

| File Suffix             | Content                                                                                              |
| ----------------------- | ---------------------------------------------------------------------------------------------------- |
| `_semantic_map.json`    | Semantic map: contains structured information such as station coordinates, approach points, and boundaries. This file is loaded when `app.py` runs. |
| `_occupancy_grid.npy`   | Occupancy grid: a NumPy array where FREE=0 / OBSTACLE=1 / STATION=2 / APPROACH=3 / ROBOT=4, used for A* path planning. |
| `_map.png`              | Visualized map image                                                                                 |

**Map generation tool** — `robosuite/robosuite/environments/factory_sorting/get_map.py`

Contestants can:

- Refer to this script to understand the map generation logic.
- Modify input parameters to regenerate the grid, such as adjusting resolution and obstacle size.
- Read the original JSON files under `maps/` in their own skills to build custom maps.

> Path: `robosuite/robosuite/environments/factory_sorting/`

Run command, using L1 as an example:

```bash
python robosuite/robosuite/environments/factory_sorting/load_factory_sorting_1_3fo3erfhisem_scene.py
```

**Trajectory collection script** — `load_factory_sorting_1_3fo3erfhisem_collect.py`

> Path: `robosuite/robosuite/environments/factory_sorting/`

This script implements a complete **robot demonstration trajectory collection workflow**:

1. Start the MuJoCo simulation scene.
2. The robot starts from the navigation pose and performs a dual-arm side-grasping action.
3. After a successful grasp, convert the trajectory data into an HDF5 file in robomimic format.
4. The generated `.hdf5` file is saved in `robosuite/robosuite/models/assets/demonstrations_private/`.

The collected trajectory data include:

- Low-dimensional observations, such as Tiago end-effector poses and gripper states.
- Image observations from the `robot0_robotview` camera.

**Contestants can:**

- Refer to the demonstration collection logic in `load_factory_sorting_1_3fo3erfhisem_collect.py`.
- Write similar collection scripts for the other four scenes. Currently, only the L1 scene collection script is provided.
- Collect grasping/placement demonstration trajectories for each scene.
- Use the collected `.hdf5` data to train their own robomimic BC policy model.
- Write the path of the trained `.pth` model into `grasp_policy.checkpoint_path` in `knowledge/robot_params.json` to replace the default policy.

### Simulation Views

The platform uses two MuJoCo camera views, which can be switched in the replay panel of `app.py`:

| View         | Camera Name          | Field of View                  | Purpose                                                                                     |
| ------------ | -------------------- | ------------------------------ | ------------------------------------------------------------------------------------------- |
| **Birdview** | `birdview`           | Top-down view of the whole factory | Navigation trajectory recording and replay, and global path visualization. This is the default view and records the complete route of the robot base movement. |
| **Robotview** | `robot0_robotview`   | Forward-facing view from the Tiago robot chest | Observation of the grasping process. The BC policy model obtains 128 x 128 RGB images from this view as visual input. During replay, it is used to inspect manipulator operation details. |

**Usage:**

- **Replay panel**: In the sidebar `Camera` drop-down menu, switch between `birdview` and `robot0_robotview`, then click `Replay -> GIF` to generate an animation from the selected view.
- **Grasp segment**: When `Replay Scope` is set to `Grasp Segment`, `robot0_robotview` is used by default to record a close-up of grasping.
- **Code call**: `RobosuiteBackend` specifies the view through the `camera` parameter, such as `RobosuiteBackend(env_name=..., camera="birdview")`.

## II. Complete Parameter Reference for `knowledge/robot_params.json`

> After modifying this file, **restart `app.py`** for the changes to take effect. No `.py` code needs to be changed.
> All numeric parameters are validated against safety ranges. Values outside the range will be automatically clamped and a warning will be printed.

### `llm` — Large Model Configuration

Controls access to the large models used for task planning and visual understanding. Parameters take effect through `config.py` -> `AgentConfig` and `app.py`. **Environment variables have higher priority than this file**.

#### Ollama (default)

| Parameter         | Type   | Default                            | Meaning                                                                |
| ----------------- | ------ | ---------------------------------- | ---------------------------------------------------------------------- |
| `ollama_base_url` | string | `"http://localhost:11434"`         | Ollama service address.                                                |
| `ollama_model`    | string | `"qwen3.6:27b-mtp-q4_K_M"`        | Text model for task planning.                                          |
| `vision_model`    | string | `"qwen3-vl:8b"`                    | Vision model for SOP document image analysis.                          |

#### OpenAI-Compatible API (DeepSeek, Zhipu GLM, etc.)

| Parameter         | Type   | Default                         | Meaning                                                                |
| ----------------- | ------ | ------------------------------- | ---------------------------------------------------------------------- |
| `openai_base_url` | string | `"https://api.deepseek.com"`    | OpenAI-compatible API endpoint.                                        |
| `openai_model`    | string | `"deepseek-v4-flash"`               | Model name to use for planning.                                        |

Environment variables (`OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL`) override these defaults.

### `navigation` — Navigation Parameters (12 parameters)

Controls the movement speed, accuracy, and drive mode of the base. All parameters take effect through `skills/move.py` -> `backend.follow_path()`.

| Parameter             | Type   | Default  | Range             | Meaning                                                                  |
| --------------------- | ------ | -------- | ----------------- | ------------------------------------------------------------------------ |
| `max_steps`           | int    | 3000     | 500-10000         | Maximum navigation steps. A larger value allows longer-distance movement but increases timeout risk. |
| `waypoint_tolerance`  | float  | 0.01     | 0.001-0.10        | Distance threshold for judging that a waypoint has been reached, in meters. Smaller values are more precise but may make the target unreachable. |
| `settle_steps`        | int    | 5        | 0-20              | Stabilization steps after reaching the endpoint. Allows the robot to stop and eliminate inertia. |
| `max_linear`          | float  | 0.70     | 0.1-2.0           | Maximum linear velocity (m/s). Higher speed increases collision risk.    |
| `max_angular`         | float  | 1.20     | 0.1-3.0           | Maximum angular velocity (rad/s). Higher values cause sharper turns.     |
| `k_linear`            | float  | 1.2      | 0.1-5.0           | Proportional gain for linear velocity. Larger values produce more aggressive acceleration. |
| `k_angular`           | float  | 1.8      | 0.1-5.0           | Proportional gain for angular velocity. Larger values produce more aggressive turning. |
| `holonomic_base`      | bool   | true     | true/false        | Whether to enable omnidirectional movement, including lateral translation. |
| `yaw_control`         | bool   | false    | true/false        | Whether to enable orientation control, actively adjusting heading during navigation. |
| `turn_in_place_angle` | float  | 0.65     | 0.1-3.0           | In-place turning threshold (rad). If the heading error exceeds this value, the robot turns before moving. |
| `drive_mode`          | string | "direct" | "direct"/"action" | Drive mode. "direct" means directly setting the position; "action" means velocity control. |
| `control_freq`        | int    | 20       | 10-50             | Control frequency (Hz). Higher values provide finer control but increase computation. |

### `grasp_policy` — BC Grasping Policy Parameters (8 parameters)

Controls neural-network-based grasp execution using Behavior Cloning (BC). All parameters take effect through `skills/pick_up.py` -> `backend.grasp_object_physics()`.

| Parameter                  | Type   | Default Value                                | Range      | Meaning                                                                  |
| -------------------------- | ------ | -------------------------------------------- | ---------- | ------------------------------------------------------------------------ |
| `checkpoint_path`          | string | `"robosuite/robosuite/model_epoch_500.pth"` | —          | BC policy model file path, relative to the project root. Contestants may replace it with their own trained model. |
| `checkpoint_fallback_path` | string | `"robosuite/robosuite/model_epoch_150.pth"` | —          | Fallback model path. Used when the file specified by `checkpoint_path` does not exist. |
| `eval_steps`               | int    | 360                                          | 100-1000   | Number of policy execution steps. The model outputs one action per step. Too few steps may fail to grasp; too many waste time. |
| `post_hold_steps`          | int    | 5                                            | 0-30       | Holding steps after grasp completion. Allows the gripper to close stably. |
| `initial_view_steps`       | int    | 5                                            | 0-30       | Observation steps before policy execution. Provides the model with initial visual input. |
| `debug_policy`             | bool   | false                                        | true/false | Whether to print policy debugging information, with diagnostics output every N steps. |
| `debug_every`              | int    | 25                                           | 1-100      | Interval for printing debugging information, in steps. Valid only when `debug_policy=true`. |
| `record_frame_interval`    | int    | 5                                            | 1-50       | Interval for recording trajectory frames, in steps. Smaller values produce denser trajectories but larger files. |

### `lift` — Lift Verification Parameters (5 parameters)

After grasping, the robot lifts the object upward to verify whether the grasp is stable. All parameters take effect through `skills/pick_up.py` -> `backend.grasp_object_physics()`.

| Parameter      | Type  | Default | Range       | Meaning                                                                  |
| -------------- | ----- | ------- | ----------- | ------------------------------------------------------------------------ |
| `lift_height`  | float | 0.15    | 0.05-0.50   | Target lift height, in meters. The object is lifted upward by this distance from its current position. |
| `max_steps`    | int   | 300     | 50-1000     | Maximum lift steps. If the target height is not reached within this number of steps, the grasp is judged as failed. |
| `hold_steps`   | int   | 20      | 5-100       | Holding steps after reaching the lift height. Verifies that the object does not slip. |
| `tolerance`    | float | 0.02    | 0.001-0.10  | Tolerance for reaching the lift height, in meters. This is the allowed error from the target height. |
| `max_action`   | float | 0.80    | 0.10-2.0    | Maximum magnitude of the lifting action. Larger values make the arm movement more aggressive. |

### `turn` — Turning Parameters (4 parameters)

Controls turning alignment so that the robot faces the target station before placement. All parameters take effect through `skills/place_down.py` -> `backend.place_object_physics()`.

| Parameter      | Type  | Default | Range       | Meaning                                                                  |
| -------------- | ----- | ------- | ----------- | ------------------------------------------------------------------------ |
| `tolerance`    | float | 0.02    | 0.001-0.10  | Turning tolerance (rad). If the heading error is smaller than this value, the robot is considered aligned. |
| `max_iters`    | int   | 8       | 1-30        | Maximum number of turning iterations. Each iteration executes `turn_steps` steps. |
| `turn_steps`   | int   | 40      | 10-200      | Number of steps for each turning iteration.                              |
| `settle_steps` | int   | 10      | 0-50        | Stabilization steps after turning is completed.                          |

### `place` — Placement Parameters (4 parameters)

Controls the fine operation of lowering and releasing the object. All parameters take effect through `skills/place_down.py` -> `backend.place_object_physics()`.

| Parameter           | Type  | Default | Range      | Meaning                                                                  |
| ------------------- | ----- | ------- | ---------- | ------------------------------------------------------------------------ |
| `lower_delta`       | float | 0.18    | 0.05-0.50  | Lowering height difference, in meters. The object is lowered by this distance from the holding height. |
| `lower_steps`       | int   | 25      | 5-100      | Number of steps for the lowering process. More steps make lowering smoother but slower. |
| `release_steps`     | int   | 60      | 10-300     | Waiting steps after release. Allows the object to naturally fall onto the table under gravity. |
| `release_clearance` | float | 0.04    | 0.01-0.20  | Release clearance, in meters. Safe distance between the object bottom and the table to avoid penetration. |

## Quick Start

```bash
# python>=3.11
cd JCIIOT2026
pip install -r requirements.txt
pip install -e ./robosuite
pip install -e .

# Ensure that the following files exist:
#   robosuite/robosuite/model_epoch_150.pth
```

### LLM Backend Configuration

The sidebar provides a **backend selector** to switch between LLM providers:

| Backend        | Configuration                                  | Example                                  |
| -------------- | ---------------------------------------------- | ---------------------------------------- |
| **Ollama**     | URL + Model name                               | `http://localhost:11434` / `qwen3.6`     |
| **OpenAI API** | API Key + Base URL + Model                     | DeepSeek, Zhipu GLM, OpenAI, vLLM, etc.  |
| **Local GGUF** | Path to `.gguf` file                           | `C:\models\qwen3.gguf`                   |

Click **"Test Connection"** to verify — a persistent status indicator (green = connected, red = failed) is shown at the top of the sidebar. During task execution, the active backend and model are displayed in the results panel and LLM Thought Process expander.

### VLM (Vision Language Model) Configuration

The sidebar **Vision Model (VLM)** section provides independent configuration for multimodal models used in document image analysis:

| Field | Description | Example |
|-------|-------------|---------|
| **VLM Base URL** | API endpoint | `http://localhost:11434` (Ollama) or `https://open.bigmodel.cn/api/paas/v4` (Zhipu) |
| **VLM API Key** | API key (leave empty for Ollama) | `sk-...` |
| **VLM Model** | Model name | `qwen3-vl:8b` / `gpt-4o` / `glm-4.6v` |

- **Status indicator** shows connection state (Connected / Disconnected / not tested).
- **"Test VLM with Factory Scene"** button loads the MuJoCo simulation, captures a birdview frame, sends it to the VLM, and displays both the image and the model's description.
- If VLM fields are left empty, the system falls back to the text LLM backend configuration or defaults to Ollama.

#### Using OpenAI-compatible VLM in skills

```python
from robot_agent.core.vision_client import ask_vision

# Ollama (default)
response = ask_vision(prompt, image_bytes,
                      base_url="http://localhost:11434",
                      model="qwen3-vl:8b")

# OpenAI-compatible (Zhipu GLM, GPT-4o, etc.)
response = ask_vision(prompt, image_bytes,
                      base_url="https://open.bigmodel.cn/api/paas/v4",
                      model="glm-4.6v",
                      api_type="openai",
                      api_key="your-api-key")

# Auto-detect API type from URL
from robot_agent.core.vision_client import ask_vision_auto
response = ask_vision_auto(prompt, image_bytes,
                           base_url="https://api.deepseek.com",
                           model="deepseek-v4-flash",
                           api_key="sk-...")
```
